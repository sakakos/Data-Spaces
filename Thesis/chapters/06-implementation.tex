\chapter{Υλοποίηση και Στήσιμο}
\label{ch:implementation}

Το παρόν κεφάλαιο περιγράφει αναλυτικά τη διαδικασία στησίματος του
\textlatin{Data Space}, από τη δημιουργία των κρυπτογραφικών πιστοποιητικών
έως την παραμετροποίηση κάθε συστατικού και τη δημιουργία πολιτικών εξουσιοδότησης.
Τα βήματα που παρουσιάζονται πρέπει να εκτελεστούν με αυτή την σειρά.

% ------------------------------------------------------------------
\section{Βήμα 1: Δημιουργία Κρυπτογραφικών Πιστοποιητικών}
\label{sec:step1-certs}
% ------------------------------------------------------------------

Το πρώτο βήμα της υλοποίησης είναι η δημιουργία της κρυπτογραφικής υποδομής
(\textlatin{PKI}) που θα χρησιμοποιηθεί για την ταυτοποίηση των συμμετεχόντων.
Απαιτούνται τρία ζεύγη κλειδιών και πιστοποιητικών: ένα για την \textlatin{CA},
ένα για τον πάροχο και ένα για τον καταναλωτή.


Το \textlatin{script} \textlatin{01\_generate\_certs.py} αυτοματοποιεί την παραγωγή
όλων των απαραίτητων πιστοποιητικών χρησιμοποιώντας τη βιβλιοθήκη
\textlatin{Python} \textlatin{cryptography}. Εκτελείται μέσω \textlatin{Docker}
ώστε να εξασφαλίζεται η αναπαραγωγιμότητα ανεξαρτήτως του τοπικού περιβάλλοντος:

\selectlanguage{english}
\begin{lstlisting}[language=bash, caption={Certificate generation script execution.}, stringstyle=\color{black}, keywordstyle=\color{black}]
docker run --rm \
  -v ".\iShare:/iShare" \
  python:3.9-slim sh -c \
  "pip install cryptography -q && python3 /iShare/01_generate_certs.py"
\end{lstlisting}

\selectlanguage{greek}
\noindent Τα ορίσματα της εντολής ερμηνεύονται ως εξής:
\begin{itemize}
    \item \textlatin{-{}-rm}: Αφαιρεί αυτόματα το \textlatin{container} μετά την
    εκτέλεση.
    \item \textlatin{-v ".\textbackslash iShare:/iShare"}: Προσαρτά τον τοπικό
    φάκελο \textlatin{iShare} στο \textlatin{container}, ώστε τα παραγόμενα αρχεία
    να αποθηκευτούν τοπικά.
    \item \textlatin{python:3.9-slim}: Χρησιμοποιεί μια ελαφριά εικόνα
    \textlatin{Python 3.9}.
    \item \textlatin{pip install cryptography}: Εγκαθιστά τη βιβλιοθήκη
    κρυπτογραφίας.
    \item \textlatin{python3 /iShare/01\_generate\_certs.py}: Εκτελεί το
    \textlatin{script}.
\end{itemize}

Το \textlatin{script} παράγει τα ακόλουθα αρχεία στον φάκελο \textlatin{iShare/}:

\begin{itemize}
    \item \textlatin{ca.key}, \textlatin{ca.pem}: Το ιδιωτικό κλειδί και
    το πιστοποιητικό της \textlatin{CA} (\textlatin{PlegmaCA}). Η \textlatin{CA}
    υπογράφει όλα τα υπόλοιπα πιστοποιητικά.
    \item \textlatin{client.key}: Το ιδιωτικό κλειδί του παρόχου σε
    μορφή \textlatin{PKCS\#8} (\textlatin{BEGIN PRIVATE KEY}), κατάλληλο για
    \textlatin{Python} και \textlatin{DSBA-PDP}.
    \item \textlatin{client\_rsa.key}: Το ίδιο κλειδί σε μορφή
    \textlatin{PKCS\#1} (\textlatin{BEGIN RSA PRIVATE KEY}), απαραίτητο για το
    \textlatin{Kong}, καθώς η βιβλιοθήκη \textlatin{Lua} \textlatin{lua-resty-jwt}
    που χρησιμοποιεί απαιτεί αυτή τη μορφή.
    \item \textlatin{client.pem}: Το πιστοποιητικό του παρόχου.
    \item \textlatin{fullchain.pem}: Η πλήρης αλυσίδα πιστοποιητικών
    παρόχου (πιστοποιητικό παρόχου + πιστοποιητικό \textlatin{CA}).
    \item \textlatin{consumer.key}, \textlatin{consumer.pem},
    \textlatin{consumer\_fullchain.pem}: Τα αντίστοιχα αρχεία για τον καταναλωτή.
\end{itemize}

Δύο παράμετροι είναι κρίσιμες για τη σωστή λειτουργία του συστήματος:

\begin{enumerate}
    \item Επέκταση \textlatin{keyUsage: digitalSignature}: Τα
    πιστοποιητικά παρόχου και καταναλωτή φέρουν υποχρεωτικά την επέκταση
    \textlatin{X.509v3} \textlatin{keyUsage} με τιμή \textlatin{digitalSignature}.
    Ο \textlatin{Keyrock} ελέγχει ρητά αυτή την επέκταση κατά την επαλήθευση
    πιστοποιητικών μέσω του \textlatin{endpoint} \textlatin{/extparticipant}.
    Απουσία αυτής της επέκτασης οδηγεί σε σφάλμα κατά την αρχική επαλήθευση
    του καταναλωτή.

    \item Αριθμός \textlatin{EORI} στο πεδίο \textlatin{Serial Number}:
    Το αναγνωριστικό \textlatin{EORI} κάθε συμμετέχοντος τοποθετείται στο
    πεδίο \textlatin{SerialNumber} του \textlatin{Distinguished Name} του
    πιστοποιητικού. Ο \textlatin{iSHARE Satellite} χρησιμοποιεί αυτό το πεδίο
    για να αντιστοιχίσει ένα πιστοποιητικό στον αντίστοιχο συμμετέχοντα.
\end{enumerate}

Μετά την εκτέλεση, το \textlatin{script} εκτυπώνει τις πληροφορίες που
απαιτούνται για την παραμετροποίηση των υπόλοιπων συστατικών:

\selectlanguage{english}
\begin{lstlisting}[language=bash, caption={Sample output of the certificate generation script.}, stringstyle=\color{black}, keywordstyle=\color{black}]
Provider fingerprint: 19A0DD7A2074F4B02E5CB34E63BD88C1...
Consumer fingerprint: B26DB03FC20A09C89070FDCE99E5E18F...
CA fingerprint:       F2CDD6D6BB356D684F9D42DC5502DFA7...

--- satellite.yml: fingerprints section ---
  Provider certificate_fingerprint: "19A0DD7A..."
  Consumer certificate_fingerprint: "B26DB03F..."

--- kong.yml: jws section ---
  x5c: "<base64_cert_provider>,<base64_CA>"
\end{lstlisting}
\selectlanguage{greek}

\section{Βήμα 2: Παραμετροποίηση \textlatin{iSHARE Satellite}}
\label{sec:step2-satellite}

Ο \textlatin{iSHARE Satellite} παραμετροποιείται μέσω του αρχείου
\textlatin{satellite.yml}. Το αρχείο αυτό ορίζει:

\begin{itemize}
    \item Το αναγνωριστικό και τα κρυπτογραφικά κλειδιά του \textlatin{Satellite}
    (ίδια με τον πάροχο, καθώς ο \textlatin{Satellite} λειτουργεί υπό την
    ταυτότητα \textlatin{EU.EORI.NLPLEGMA}).
    \item Τη λίστα εμπιστευμένων \textlatin{CA} (\textlatin{trusted\_list}),
    που περιλαμβάνει το αποτύπωμα της \textlatin{PlegmaCA}.
    \item Το μητρώο εγκεκριμένων συμμετεχόντων (\textlatin{parties}), με εγγραφές
    για τον πάροχο και τον καταναλωτή:
\end{itemize}

\selectlanguage{english}
\begin{lstlisting}[language=bash, caption={Excerpt of satellite.yml for participant registration.}, stringstyle=\color{black}, keywordstyle=\color{black}]
parties:
  - id: EU.EORI.NLPLEGMA
    name: Plegma Data Provider
    status: Active
    certifications:
      - role: IdentityProvider
      - role: AuthorisationRegistry
    certificates:
      - certificate_fingerprint: "19A0DD7A..."
        enabled_from: "2026-01-01T00:00:00Z"

  - id: EU.EORI.NLPLEGMACONSUMER
    name: Plegma Data Consumer
    status: Active
    certifications:
      - role: ServiceConsumer
    certificates:
      - certificate_fingerprint: "B26DB03F..."
        enabled_from: "2026-01-01T00:00:00Z"
\end{lstlisting}
\selectlanguage{greek}

\section{Βήμα 3: Παραμετροποίηση \textlatin{Kong API Gateway}}
\label{sec:step3-kong}
% ------------------------------------------------------------------

Το \textlatin{Kong} παραμετροποιείται μέσω του αρχείου \textlatin{kong.yml} σε
δηλωτική μορφή (\textlatin{declarative configuration}). Ορίζεται μια υπηρεσία
(\textlatin{energy-data-service}) που δρομολογεί τις αιτήσεις προς τον
\textlatin{Orion-LD}, και προσαρτάται το \textlatin{plugin}
\textlatin{ngsi-ishare-policies} με τις ακόλουθες παραμέτρους:

\selectlanguage{english}
\begin{lstlisting}[language=bash, caption={Excerpt of kong.yml -- plugin configuration.}]
plugins:
  - name: ngsi-ishare-policies
    config:
      jws:
        identifier: EU.EORI.NLPLEGMA
        root_ca_file: /iShare/certificate.pem
        private_key: /iShare/key.pem     # Mounted: client_rsa.key
        x5c: "<cert_base64>,<CA_base64>" # From 01_generate_certs.py
      satellite:
        identifier: EU.EORI.NLPLEGMA
        host: http://ishare-satellite:8080
        token_endpoint: /token
        trusted_list_endpoint: /trusted_list
      ar:
        identifier: EU.EORI.NLPLEGMA
        host: http://fiware-keyrock:3007
        token_endpoint: /oauth2/token
        delegation_endpoint: /ar/delegation
\end{lstlisting}
\selectlanguage{greek}

Κρίσιμη λεπτομέρεια είναι ότι το ιδιωτικό κλειδί που προσαρτάται στο
\textlatin{Kong} είναι το \textlatin{client\_rsa.key} (μορφή \textlatin{PKCS\#1})
και όχι το \textlatin{client.key} (μορφή \textlatin{PKCS\#8}). Η βιβλιοθήκη
\textlatin{Lua} \textlatin{lua-resty-jwt} που χρησιμοποιεί εσωτερικά το
\textlatin{Kong} για τη δημιουργία \textlatin{JWT} κατά την επικοινωνία με τον
\textlatin{Satellite} απαιτεί πλαίσιο \textlatin{BEGIN RSA PRIVATE KEY} και όχι
\textlatin{BEGIN PRIVATE KEY}.

\section{Βήμα 4: Παραμετροποίηση \textlatin{Keyrock}}
\label{sec:step4-keyrock}
% ------------------------------------------------------------------

Ο \textlatin{Keyrock} παραμετροποιείται μέσω μεταβλητών περιβάλλοντος στο
αρχείο \textlatin{docker-compose.yml}:

\selectlanguage{english}
\begin{lstlisting}[language=bash, caption={Keyrock environment variables for iSHARE.}, stringstyle=\color{black}, keywordstyle=\color{black}]
environment:
  - IDM_PR_CLIENT_ID=EU.EORI.NLPLEGMA
  - IDM_PR_ID=EU.EORI.NLPLEGMA
  - IDM_PR_URL=http://ishare-satellite:8080
  - IDM_PR_TOKEN_ENDPOINT=http://ishare-satellite:8080/token
  - IDM_PR_PARTIES_ENDPOINT=http://ishare-satellite:8080/parties
  - IDM_PR_CLIENT_KEY=/iShare/key.pem
  - IDM_PR_CLIENT_CRT=/iShare/certificate.pem
  - IDM_AR_ID=EU.EORI.NLPLEGMA
  - IDM_AR_URL=internal
  - IDM_AR_TOKEN_ENDPOINT=http://fiware-keyrock:3007/oauth2/token
  - IDM_AR_DELEGATION_ENDPOINT=http://fiware-keyrock:3007/ar/delegation
  - IDM_PDP_LEVEL=advanced
\end{lstlisting}
\selectlanguage{greek}
\noindent Οι παράμετροι αυτές ορίζουν:
\begin{itemize}
    \item \textlatin{IDM\_PR\_*}: Παράμετροι για τη σύνδεση του \textlatin{Keyrock}
    με τον \textlatin{iSHARE Satellite} (\textlatin{PR} = \textlatin{Participant
    Registry}).
    \item \textlatin{IDM\_AR\_*}: Παράμετροι που ορίζουν τον \textlatin{Keyrock}
    ως εσωτερικό \textlatin{Authorization Registry} (\textlatin{IDM\_AR\_URL=internal}).
    \item \textlatin{IDM\_PDP\_LEVEL=advanced}: Ενεργοποιεί τον έλεγχο πρόσβασης
    βάσει πολιτικών \textlatin{iSHARE}.
\end{itemize}

Η έκδοση του \textlatin{Keyrock} που χρησιμοποιείται απαιτεί δύο διορθώσεις
στον πηγαίο κώδικα, οι οποίες εφαρμόζονται αυτόματα μέσω του \textlatin{script}
\textlatin{02\_setup\_keyrock.py}:

\begin{enumerate}
    \item \textbf{Patch \textlatin{model\_oauth\_server.js}}: Προσθέτει το πεδίο
    \textlatin{admin} στη λίστα \textlatin{attributes} που ανακτώνται κατά τον
    έλεγχο ταυτότητας. Χωρίς αυτή τη διόρθωση, τα \textlatin{tokens} που
    εκδίδει ο \textlatin{Keyrock} δεν περιέχουν τα απαραίτητα δεδομένα
    \textlatin{delegation evidence}.

    \item \textbf{Patch \textlatin{configService.js}}: Τροποποιεί τον τρόπο με τον
    οποίο ο \textlatin{Keyrock} φορτώνει τα κρυπτογραφικά κλειδιά, επιτρέποντας
    τη χρήση διαδρομής αρχείου αντί για απευθείας τιμή \textlatin{string}. Αυτό
    επιτρέπει τη φόρτωση των κλειδιών μέσω \textlatin{Docker volumes} χρησιμοποιώντας
    τις μεταβλητές περιβάλλοντος \textlatin{IDM\_PR\_CLIENT\_KEY} και
    \textlatin{IDM\_PR\_CLIENT\_CRT}.
\end{enumerate}

\subsection{Αρχική Καταχώρηση Καταναλωτή}
\label{subsec:consumer-registration}

Πριν από την εκτέλεση του \textlatin{script} παραμετροποίησης, ο καταναλωτής
πρέπει να έχει πραγματοποιήσει τουλάχιστον ένα αίτημα \textlatin{token} προς
τον \textlatin{Keyrock}. Αυτό είναι απαραίτητο διότι ο \textlatin{Keyrock} δεν
δημιουργεί χρήστες για εξωτερικούς συμμετέχοντες \textlatin{iSHARE} μέχρι την
πρώτη τους επικοινωνία. Χωρίς την εγγραφή αυτή, το \textlatin{UUID} του
καταναλωτή δεν υπάρχει στη βάση δεδομένων και η πολιτική εκχώρησης δεν μπορεί
να αποθηκευτεί.

Η αρχική καταχώρηση γίνεται με την ακόλουθη ακολουθία εντολών:

\selectlanguage{english}
\begin{lstlisting}[language=bash, caption={Initial consumer registration in Keyrock.}, stringstyle=\color{black}, keywordstyle=\color{black}]
CONSUMER_JWT=$(docker run --rm \
  -v ".\iShare:/iShare" python:3.9-slim sh -c \
  "pip install PyJWT cryptography -q && \
   python3 /iShare/gen_consumer_token.py")

curl -s -X POST "http://localhost:3007/oauth2/token" \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "grant_type=client_credentials \
      &scope=iSHARE \
      &client_assertion_type=urn:ietf:params:oauth:client-assertion-type:jwt-bearer \
      &client_assertion=${CONSUMER_JWT} \
      &client_id=EU.EORI.NLPLEGMACONSUMER"
\end{lstlisting}
\selectlanguage{greek}

\noindent Τα ορίσματα του αιτήματος \textlatin{OAuth 2.0} ερμηνεύονται ως εξής:
\begin{itemize}
    \item \textlatin{grant\_type=client\_credentials}: Ροή \textlatin{M2M} χωρίς
    εμπλοκή χρήστη.
    \item \textlatin{scope=iSHARE}: Υποδηλώνει ότι το αίτημα ακολουθεί το
    πρωτόκολλο \textlatin{iSHARE}.
    \item \textlatin{client\_assertion\_type=...jwt-bearer}: Ο τύπος \textlatin{assertion}
    που χρησιμοποιείται, \textlatin{JWT} αντί για παραδοσιακό \textlatin{secret}.
    \item \textlatin{client\_assertion}: Το ίδιο το \textlatin{iSHARE JWT} του
    καταναλωτή.
    \item \textlatin{client\_id}: Το \textlatin{EORI} του καταναλωτή.
\end{itemize}

Η πολιτική που δημιουργείται επιτρέπει στον καταναλωτή να εκτελεί λειτουργίες
\textlatin{GET} σε οποιοδήποτε \textlatin{entity} τύπου
\textlatin{HouseholdElectricMeasurement}, ισχύει από το 2026 έως το 2027 και έχει
εκδοθεί από τον πάροχο \textlatin{EU.EORI.NLPLEGMA}.

Κρίσιμη λεπτομέρεια της υλοποίησης είναι ότι η πολιτική αποθηκεύεται με
αναφορά στο εσωτερικό \textlatin{UUID} του καταναλωτή στη βάση δεδομένων του
\textlatin{Keyrock} (π.χ. \textlatin{f9b23a01-a381-458b-9ae4-ff9d7a86ba0f}) και
όχι στο \textlatin{EORI} του. Αυτό συμβαίνει διότι η εσωτερική λογική του
\textlatin{Keyrock} αναζητά τις πολιτικές βάσει \textlatin{UUID}, όχι βάσει
\textlatin{EORI}. Το \textlatin{script} \textlatin{02\_setup\_keyrock.py} αναλαμβάνει
αυτόματα την ανάκτηση του σωστού \textlatin{UUID} από τη βάση δεδομένων
\textlatin{MySQL}.

% ------------------------------------------------------------------
\section{Βήμα 7: Φόρτωση Δεδομένων \textlatin{Plegma Dataset}}
\label{sec:step7-data}
% ------------------------------------------------------------------

Μετά την ολοκλήρωση του στησίματος, φορτώνονται τα δεδομένα του
\textlatin{Plegma Dataset}~\cite{plegma2024} στον \textlatin{Orion-LD} μέσω
του \textlatin{script} \textlatin{orion\_ld\_loading\_script\_kong.py}. Το \textlatin{script}
διαβάζει τα δεδομένα από \textlatin{CSV} αρχεία και τα αποστέλλει στον
\textlatin{Orion-LD} μέσω του \textlatin{Kong}, εξασφαλίζοντας ότι και η εγγραφή
δεδομένων ακολουθεί τους μηχανισμούς ελέγχου πρόσβασης:

\selectlanguage{english}
\begin{lstlisting}[language=bash, caption={Data loading via Kong.}]
python3 orion_ld_loading_script_kong.py
\end{lstlisting}
\selectlanguage{greek}
Το \textlatin{script} αυτό δημιουργεί για κάθε γραμμή του \textlatin{CSV}:
\begin{enumerate}
    \item Ένα \textlatin{iSHARE JWT} με 30 δευτερόλεπτα διάρκεια ισχύος,
    υπογεγραμμένο με το ιδιωτικό κλειδί του παρόχου.
    \item Ένα αίτημα \textlatin{HTTP POST} προς το
    \textlatin{http://localhost:8000/ngsi-ld/v1/entities} με το \textlatin{entity}
    σε μορφή \textlatin{JSON-LD} και το \textlatin{JWT} ως \textlatin{Bearer token}.
\end{enumerate}

Η αιτιολογία για τη δημιουργία νέου \textlatin{JWT} ανά \textlatin{entity} έγκειται
στη σύντομη διάρκεια ισχύος των \textlatin{iSHARE JWT} (30 δευτερόλεπτα): η
φόρτωση πολλών εγγραφών μπορεί να υπερβαίνει αυτό το χρονικό παράθυρο, οπότε
κάθε αίτημα πρέπει να φέρει φρέσκο \textlatin{token}.

% ------------------------------------------------------------------
\section{Σύνοψη Διαδικασίας Στησίματος}
\label{sec:setup-summary}
% ------------------------------------------------------------------

Η διαδικασία στησίματος του \textlatin{Plegma Data Space} συνοψίζεται στα
ακόλουθα επτά βήματα:

\begin{enumerate}
    \item Δημιουργία κρυπτογραφικών πιστοποιητικών
    (\textlatin{01\_generate\_certs.py}).
    \item Παραμετροποίηση \textlatin{iSHARE Satellite} (\textlatin{satellite.yml}).
    \item Παραμετροποίηση \textlatin{Kong} (\textlatin{kong.yml}).
